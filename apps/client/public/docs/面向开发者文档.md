# 面向开发者文档

本文档面向未来维护本项目的人类开发者，也可作为 AI 的技术背景资料。它描述的是工程架构、实现方式、模块职责与维护边界，而不是给玩家看的玩法说明。

## 1. 项目定位

`刺杀苏丹王 KillSultan` 是一个基于网页的多人实时身份推理游戏。

当前工程目标不是“单页网页展示”，而是一套完整的小型多人游戏系统：

- 前端负责界面、交互、动画和玩家体验。
- Node 后端负责房间管理、Socket 连接、状态广播和外围适配。
- `apps/core_rules` 负责规则判定、回合推进和结算真值。
- shared 负责前后端共用的类型、协议与规则辅助函数。
- docs 负责玩家说明、开发说明、AI 交接与项目治理。

## 1.1 维护边界补充

`apps/core_rules` 是核心机制目录，默认由开发者 `zhz` 维护。

对开发者 `lxh`、以及未来参与本项目的 AI 来说，应遵守以下边界：

- 除非 `lxh` 明确说明，否则不要主动改动 `apps/core_rules` 内的文件。
- AI 默认只能把 `apps/core_rules` 视为“被保护的规则真值层”，不应擅自重写、重构或替换。
- 动画、通信、数据库、前端表现、外层服务适配等工作，应优先在 `apps/core_rules` 之外完成。
- 若必须调整核心规则接口，应先与 `zhz` 对齐边界，再进行改动。

## 2. 技术栈

当前实现主要使用 `TypeScript`。

- 前端：`React + Vite + TypeScript`
- 服务端适配层：`Node.js + Express + Socket.IO + TypeScript`
- 核心机制层：`Python`
- 共享层：`packages/shared`
- 存储：默认内存，支持 `Redis`

根目录 workspace 由 [package.json](../package.json) 管理，`npm run dev` 会同时拉起 shared、server、client。

## 4. 当前架构思想

### 4.1 规则真值与适配分层

当前架构是“Node 适配层 + Python 规则真值层”。

前端不能决定规则是否成立，只能发起动作请求；真正的判定、执行和结算由 `apps/core_rules` 完成，Node 服务端只负责外围适配。这样做的原因是：

- 防止多人局状态不一致
- 防止私密信息泄露
- 防止前端绕过规则
- 让规则真值集中，方便由人类开发者审核与维护

### 4.2 前后端共享协议

项目没有让前后端各写一套字段，而是通过 `packages/shared` 统一数据结构和事件协议。

这一层的职责包括：

- 定义游戏状态结构
- 定义玩家公开视图和私有视图
- 定义 Socket 事件协议
- 放置阵营/相邻座位等规则辅助函数

这能减少“前端以为字段叫 A，后端实际发的是 B”这类问题。

## 5. 前端实现

前端目录：`apps/client`

### 5.1 主要职责

- 连接服务端 Socket
- 创建房间 / 加入房间 / 准备 / 开局
- 展示圆桌、玩家座位、公共日志和私密结果
- 提供身份技能按钮和目标选择
- 展示动作动画、特效与状态提示
- 提供测试实验台入口

### 5.2 核心文件

- `src/main.tsx`：前端入口
- `src/App.tsx`：正式玩家界面
- `src/TestLab.tsx`：单窗口多玩家实验台
- `src/styles.css`：主要样式与动画

### 5.3 前端与后端如何配合

一次典型动作链路如下：

1. 玩家点击按钮
2. 前端通过 Socket.IO 发消息给服务端
3. Node 服务端把请求适配成核心机制接口输入
4. Python `core_rules` 判定是否合法并返回新状态
5. 服务端广播新的公共状态
6. 服务端单独推送私密结果给应该知道的人
7. 前端收到状态后重新渲染

因此，前端本质上是“展示层 + 交互层”，不是规则真值来源。

## 6. 后端实现

后端目录：`apps/server`

### 6.1 主要职责

- 房间创建、加入、离开、重连
- 调用 Python 核心机制
- 公共日志和私密反馈生成
- 状态保存与恢复

### 6.2 关键文件

- `src/index.ts`：服务端入口，挂载 Express 与 Socket.IO
- `src/game/engine.ts`：Node 适配层，连接房间管理与 Python 核心机制
- `src/game/core-rules-bridge.ts`：Node 调用 Python 核心机制的桥接层
- `src/game/setup.ts`：牌组配置与默认设置
- `src/store/memory-store.ts`：内存存储
- `src/store/redis-store.ts`：Redis 存储

### 6.3 为什么 engine.ts 仍然重要

迁移后，`engine.ts` 不再是规则真值本身，而是“房间与连接控制器”。它负责：

- 识别当前请求来自哪个玩家
- 把状态和动作打包成跨语言请求
- 调用 `apps/core_rules`
- 将结果重新接回现有 Socket 广播流程

如果前端是舞台，那么 `engine.ts` 更像主持人与调度层；真正的裁判真值在 `apps/core_rules`。

## 6.4 核心机制层实现

核心机制目录：`apps/core_rules`

### 6.4.1 主要职责

- 发牌
- 行动合法性判定
- 技能执行
- 回合推进
- 胜负结算
- 核心机制接口

### 6.4.2 当前文件

- `cli.py`：命令行入口，负责 JSON 输入输出
- `engine.py`：核心规则执行器
- `helpers.py`：规则辅助函数
- `setup.py`：牌组与卡牌初始化
- `errors.py`：统一错误定义
- `README.md`：目录级维护边界说明

### 6.4.3 维护约定

- `apps/core_rules` 默认由 `zhz` 维护。
- `lxh` 后续默认不直接改动该目录，除非明确说明。
- AI 默认不应主动改动该目录，除非 `lxh` 明确授权当前任务可以修改。

## 7. shared 实现

shared 目录：`packages/shared`

### 7.1 主要职责

- 游戏主数据结构
- Socket 事件协议
- 可公开状态与私有状态的构造
- 阵营判断、相邻座位等通用规则函数

### 7.2 关键文件

- `src/game-types.ts`
- `src/protocol.ts`
- `src/visibility.ts`
- `src/rules.ts`

### 7.3 为什么 shared 很关键

shared 是“前后端共同的语言层”。

没有这一层，项目就容易出现：

- 前后端字段不一致
- 协议升级遗漏
- 某个规则改了，只改了一边

## 9. 关于“AI 不能单独定义核心规则”的原则

这是本项目非常重要的工程原则。

### 9.1 原因

与游戏规则、判定、结算相关的代码，风险很高：

- AI 能写代码，但未必真正验证了玩法
- AI 容易把“听起来合理”当成“规则正确”
- 多阶段技能、链式效果、回合推进这类问题尤其容易出错

所以，凡是涉及：

- 核心规则真值
- 判定优先级
- 胜负条件
- 回合推进
- 多角色交互

都不应默认让 AI 单独决定，应至少由人类开发者检查、测试和拍板。

### 9.2 当前建议

AI 适合承担：

- 前端界面
- 动画与表现
- 文档整理
- 工具脚本
- 类型整理
- 非核心重构
- 测试辅助

人类开发者应优先负责：

- 规则定义
- 判定顺序
- 终局结算
- 核心机制改动审批

## 10. 是否可以把核心机制拆到 Python

可以，但这不是“小改”，而是架构级调整。

### 10.1 可行性

从技术上说完全可行。当前工程已经把最核心的规则判定迁入 `apps/core_rules`，作为单独的 Python 模块，由人类开发者主导维护。

### 10.2 这样做的价值

- 把“规则真值”从 UI 和 AI 辅助开发中隔离出来
- 让核心裁判逻辑有更清晰的所有权
- 未来更容易做单元测试、回放和独立验证

### 10.3 代价

如果把整个服务端、连接层、前端协议层全部换成 Python，代价仍然很高：

- 需要重写 Socket 与服务层
- 需要重写 shared 协议对接
- 当前 shared 是 TypeScript，不能直接被 Python 复用
- 需要重新建立前后端一致性机制

### 10.4 更稳的做法

当前已经采取的方式是：

1. 把“规则真值”和“表现层逻辑”在文档上彻底分清
2. 把核心规则迁入 `apps/core_rules`
3. 保留 Node 服务端作为外围适配层

### 10.5 当前维护约定

如果 `apps/core_rules` 已经建立并交由 `zhz` 维护，那么后续默认约定是：

- `apps/core_rules` 里的核心规则实现，以 `zhz` 为最终责任人。
- `lxh` 后续开发默认不直接改动这个目录，除非明确决定要改。
- AI 只能在 `lxh` 明确授权时协助修改该目录，否则只允许修改外围适配与表现层。

### 10.6 当前建议结论

“核心机制单独放到一个由人类维护的目录或服务里”这个思想是合理的。

但在真正迁移到 Python 之前，建议先完成两件事：

- 把规则边界文档化
- 把状态机行为做更系统的人类测试

## 11. 开发者协作建议

- 玩家规则变更时，先改规则来源，再改 UI 文案
- 对核心机制变更，要同时更新：
  - 服务端实现
  - 玩家规则手册
  - 面向开发者文档
  - AI 交接记忆库中的相关约定
- 任何“AI 生成但未经人类验证”的核心规则逻辑，都不应直接视为最终版本
