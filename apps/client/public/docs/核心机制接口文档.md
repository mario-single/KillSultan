# 核心机制接口文档

本文档记录 `apps/core_rules` 与外部适配层之间的接口约定。

## 1. 文档形式说明

本接口文档以 `Markdown` 作为唯一正式版本。

当前不建议为它额外维护 `DOCX` 版本，原因是：

- 它是开发者接口文档，不是玩家文档
- 接口定义需要频繁 diff、审阅和版本控制
- Markdown 更适合和代码一起维护

如果未来确实需要对外分发可打印版，再考虑从 Markdown 派生 DOCX，而不是双份手工维护。

## 2. 核心机制目录

- 路径：`apps/core_rules`
- 语言：`Python`
- 职责：
  - 规则真值
  - 判定
  - 回合推进
  - 胜负结算

## 3. 维护边界

- `apps/core_rules` 默认由开发者 `zhz` 维护。
- 开发者 `lxh` 后续默认不直接改动该目录，除非明确说明。
- AI 默认不能主动修改该目录，只能在 `lxh` 明确授权时辅助修改。
- 其他代码（前端、通信、动画、数据库、服务适配等）可以主要由 AI 协助完成，但应通过接口访问核心机制。
- 涉及 `apps/core_rules` 的接口调整，默认应先与 `zhz` 对齐。

## 4. 当前跨语言调用方式

当前建议采用：

- Python 作为核心机制实现
- Node/TypeScript 服务端作为外围适配层
- 通过稳定的命令接口传递 JSON 输入与输出

也就是说：

- TypeScript 不直接重写规则
- Python 不直接处理 Socket 连接和页面渲染

## 5. 接口原则

外围系统访问核心机制时，应尽量满足：

- 输入使用纯 JSON 结构
- 输出使用纯 JSON 结构
- 错误使用统一的 `code + message`
- 核心机制不依赖前端页面细节
- 核心机制不依赖数据库实现细节

## 6. 推荐输入输出结构

推荐请求结构：

```json
{
  "command": "action_reveal",
  "state": {},
  "actorId": "player-id",
  "payload": {}
}
```

推荐成功响应结构：

```json
{
  "ok": true,
  "data": {
    "state": {},
    "privateNotice": {
      "message": "..."
    }
  }
}
```

推荐失败响应结构：

```json
{
  "ok": false,
  "error": {
    "code": "INVALID_TARGET",
    "message": "目标已经死亡。"
  }
}
```

## 6.1 当前命令集合

当前 `apps/core_rules` 已暴露的命令包括：

- `start_game`
- `action_peek`
- `action_swap`
- `action_swap_center`
- `action_reveal`
- `action_decline_follow`
- `action_oracle_prediction`
- `action_end_turn`
- `disconnect`

其中：

- `action_oracle_prediction`
  - 用于占卜师看完三名玩家身份后，公开预言阵营
- `action_end_turn`
  - 目前主要用于奴隶起义连锁完成后，由起义发起者手动结束回合

## 7. 外围系统应负责什么

外围适配层可以负责：

- 房间与连接管理
- Socket 事件收发
- 前端状态展示
- 动画与表现
- 数据库存储
- 运营后台与战绩统计

但不应擅自改写：

- 核心判定真值
- 回合推进逻辑
- 胜负条件
- 多角色交互优先级

## 8. 变更流程建议

如果未来需要改核心机制：

1. 先由 `zhz` 明确规则目标，或由 `lxh` 与 `zhz` 对齐后再改
2. 再修改 `apps/core_rules`
3. 然后由外围适配层跟进接口
4. 最后同步更新：
   - 《面向开发者文档》
   - 《AI交接记忆库》
   - 《玩家规则手册》（若玩家规则受影响）
